# 描画機能仕様

## 1. 共通要件

すべての描画機能は、[フレームバッファ仕様](framebuffer.md)で定義したRenderTargetへ描画します。
座標はRenderTargetのローカル座標で指定し、描画先の境界でクリッピングします。

描画コードはmacOS版とRP2350版で共通とします。SDLの描画機能を代替実装として使用しては
いけません。RP2350では、本仕様に含まれる描画関数をすべてSRAMへ配置します。

## 2. BG描画

### 2.1 タイルと仮想画面

- BGは固定サイズのタイルで構成する
- タイルの幅と高さは、それぞれ独立した8ピクセル以上の2の冪とする
- タイルの画素は8ビットの色インデックスで表す
- BGタイルには透明色を設けず、インデックス0も黒として描画する
- タイル番号は符号なし8ビット整数とする
- 仮想画面の幅と高さはタイル単位で指定する
- 仮想画面の大きさは任意とする

仮想画面の外側は、色インデックス0で埋められているものとして描画します。仮想画面の
ループ処理は提供しません。

BGは次の構造体で指定します。

```cpp
struct Background {
    std::uint8_t tileWidth;
    std::uint8_t tileHeight;
    std::uint16_t width;
    std::uint16_t height;
    const std::uint8_t* tilemap;
    const std::uint8_t* patterns;
    std::uint8_t tileIndexMask;
};
```

`width`と`height`はタイル単位です。`tilemap`は行優先の8ビットタイル番号配列、
`patterns`はタイル番号順に並べた行優先の8ビット画素データです。
`tileWidth`と`tileHeight`は8、16、32、64、128のいずれかとします。タイル位置は除算ではなく
ビットシフト、タイル内位置はビットマスクで算出します。

`tilemap`の各値には、タイル番号以外の属性ビットを含められます。描画時のタイル番号は、
マップ値と`tileIndexMask`のANDで抽出します。`tileIndexMask`のデフォルト値は`0xff`です。
例えば`0x3f`を指定した場合は下位6ビットをタイル番号、上位2ビットをコリジョンなどの
アプリケーション固有情報として使用できます。Pixel Twinsは属性ビットの意味を解釈しません。

### 2.2 スクロールと描画

仮想画面上の任意の位置を、ピクセル単位のオフセットでRenderTargetへ描画できます。
オフセットには負の値を許可します。

```cpp
drawBackground(target, background, sourceX, sourceY);
```

RenderTargetのローカル座標`(0, 0)`には、仮想画面の`(sourceX, sourceY)`が描画されます。

RenderTarget境界に掛からない連続領域では、横方向に8画素展開した高速処理を使用します。
境界に掛かる部分だけをクリッピング対応処理で描画します。内側の領域で画素ごとの境界判定を
行ってはいけません。

## 3. 通常スプライト描画

### 3.1 データ構造

```cpp
struct Sprite {
    std::int16_t dx;
    std::int16_t dy;
    std::uint8_t sw;
    std::uint8_t sh;
    std::uint16_t _next;
    const std::uint8_t* p;
};
```

- `dx`, `dy`：RenderTarget上の描画先左上座標
- `sw`, `sh`：スプライトパターンの幅と高さ
- `_next`：同一バケット内の次のスプライトのインデックス。Pixel Twinsが内部で管理する
- `p`：行優先で格納された8ビットスプライトパターン

`_next`はコンストラクタが自動的に`0xffff`で初期化します。アプリケーションから値を指定する
必要はありません。インデックス0の画素は透明として書き込みをスキップします。

アニメーション、フレーム選択、表示時間の管理は提供せず、アプリケーションの責任とします。

### 3.2 高速描画

スプライトの高さ方向はループ処理とし、横方向を展開します。

- 頻出する幅8ピクセルには専用の高速版を用意する
- 幅16〜24ピクセルには、1ピクセル刻みで専用の高速版を用意する
- それ以外の幅には、横8画素展開と端数処理を組み合わせた汎用版を使用する
- スプライト全体がRenderTarget内にある場合は高速版を使用する
- 境界に掛かる場合はクリッピング対応版を使用する
- 完全に境界外の場合は描画しない

高速版の内側で画素ごとの境界判定を行ってはいけません。

## 4. 拡大縮小付きスプライト描画

```cpp
struct SpriteEx {
    std::int16_t dx;
    std::int16_t dy;
    std::uint8_t dw;
    std::uint8_t dh;
    std::uint8_t sw;
    std::uint8_t sh;
    std::uint16_t _next;
    const std::uint8_t* p;
};
```

- `dx`, `dy`：RenderTarget上の描画先左上座標
- `dw`, `dh`：描画後の幅と高さ
- `sw`, `sh`：元パターンの幅と高さ
- `_next`：同一バケット内の次のスプライトのインデックス。Pixel Twinsが内部で管理する
- `p`：行優先で格納された8ビットスプライトパターン

左上を基準に、最近傍法で拡大縮小します。インデックス0は透明色です。
`dw == 0`または`dh == 0`の場合は描画しません。

通常スプライトと同様にRenderTarget境界でクリッピングします。初期実装では通常スプライトほど
積極的な最適化を行わず、多用しない機能として扱います。

## 5. プリミティブ描画

プリミティブの色は8ビットの色インデックスで指定します。以下の機能を提供します。

- 塗りつぶし矩形
- 枠線矩形
- ライン
- 塗りつぶし三角形
- 塗りつぶし円および楕円
- 枠線円および楕円

すべてのプリミティブをRenderTarget境界でクリッピングします。ラインの終点を含むかどうかは
APIの保証事項とせず、実装方式に従います。ただし、macOS版とRP2350版で同じ共通実装を
使用するため、両環境の描画結果は一致しなければなりません。

座標は符号付き16ビット、幅、高さ、半径は符号なし16ビットで指定します。幅または高さが0の
矩形は描画しません。楕円の半径は0〜32767を有効範囲とし、それを超える指定は描画しません。

- 矩形の横線は8画素単位で展開する
- ラインは画面内へクリッピングしてから整数Bresenham法で描画する
- 三角形は増分エッジ関数を使用し、浮動小数点演算を行わない
- 塗り楕円は平方根や浮動小数点演算を使わない走査線方式とする
- 枠線楕円は整数の中点アルゴリズムを使用する
- 半径Xまたは半径Yが0の楕円は直線として描画する

## 6. フォント描画

フォントは8×9ピクセルの固定幅グリフとし、各行を縁と文字本体の2枚の1ビットマスクで
保持します。最上位ビットが左端の画素に対応します。

```cpp
struct GlyphRow {
    std::uint8_t outline;
    std::uint8_t body;
};

struct Glyph {
    std::array<GlyphRow, 9> rows;
};
```

1グリフは18バイトです。印字可能ASCII 95文字では合計1,710バイトになります。

```cpp
struct BitmapFont {
    const Glyph* glyphs;
    std::uint8_t firstCodepoint;
    std::uint8_t glyphCount;
    std::uint8_t fallbackCodepoint;
    ColorIndex outlineColor;
};
```

縁色はフォントに固定し、文字本体の色は描画時に指定します。本体と縁のビットが重複した場合は、
文字本体の色を優先します。範囲外の文字は`fallbackCodepoint`へ置換し、フォールバック文字も
範囲外の場合は描画しません。

文字列描画では、符号付き16ビットのstrideを指定できます。デフォルト値は8です。8未満なら
文字を詰め、8より大きければ字間を空け、負値なら右から左へ配置できます。改行では描画開始Xへ
戻り、Yを9ピクセル進めます。キャリッジリターンは無視します。タブ、自動折り返し、整数倍拡大は
提供しません。

グリフ全体がRenderTarget内にある場合は、横8画素を展開した高速版を使用します。境界に掛かる
場合だけクリッピング版を使用します。

## 7. Yバケット

実行時の比較ソートは行わず、180本のバケットを使用します。120ピクセルの画面高に加えて、
雷など縦に長いエフェクトのために下側へ60本の余裕を持たせます。

```cpp
constexpr std::uint16_t kInvalidSpriteIndex = 0xffff;
constexpr std::size_t kBucketCount = 120 + 60;

struct Bucket {
    std::uint16_t spriteTop;
    std::uint16_t spriteExTop;
};
```

通常スプライトと拡大縮小付きスプライトは別のリストとして保持します。同じバケット内で、
両種類をまたいだ相対的な描画順は保証しません。

Yソートの基準は原則としてスプライトの下端ですが、どのバケットへ登録するかは
アプリケーションの責任とします。Pixel Twinsは指定されたバケット番号へ登録し、若い番号から
順に描画します。

## 8. 固定長アロケーター

SpriteとSpriteExは、実行中に容量が変化しない固定長アロケーターへ格納します。

- 容量はアプリケーションがビルド時または初期化時に明示する
- 実行中にヒープ確保や再確保を行わない
- オブジェクトは16ビットのインデックスで参照する
- 容量超過は検出し、既存領域を破壊しない
- リスト終端および無効インデックスには`0xffff`を使用する

RP2350上の構造体配置を基準とします。macOSでも同じフィールドを持つ構造体を使用しますが、
ポインタ幅の違いによる`sizeof`の差は許容します。バイナリ保存形式として構造体を直接使用しては
いけません。

## 9. SRAM配置

RP2350では、次のコードをすべてSRAMへ配置します。

- BG描画
- 通常スプライト描画
- 拡大縮小付きスプライト描画
- プリミティブ描画
- フォント描画
- Yバケットの登録および描画
- フレームバッファのフリップ
- LED転送データの生成
- LED転送に関係する処理

macOS向けビルドではSRAM配置指定を空の属性として扱います。
