# フレームバッファ仕様

## 1. 概要

Pixel Twinsの描画系は、256色のパレット付きフレームバッファを使用します。

フレームバッファ生成まではmacOS版とマイコン版で同一のC++コードを使用します。
macOS版では完成したフレームバッファをSDL経由で表示し、マイコン版では将来実装する
LEDパネルドライバへ転送します。

この形式はLEDパネル向け転送データを生成する際の基準形式であり、プラットフォーム固有の
表示実装で別形式のフレームバッファを生成してはいけません。

## 2. 論理画面

物理画面は、160×120ピクセルのLEDパネル2枚で構成します。
論理フレームバッファ上では2枚を横方向に連続して配置し、320×120ピクセルの1画面として
扱います。

```text
               論理フレームバッファ 320×120
+-------------------------------+-------------------------------+
|                               |                               |
|       左パネル 160×120        |       右パネル 160×120        |
|       x = 0〜159              |       x = 160〜319            |
|                               |                               |
+-------------------------------+-------------------------------+
```

座標の原点は左上です。x座標は右方向、y座標は下方向へ増加します。

| 項目 | 値 |
| --- | ---: |
| 論理画面幅 | 320ピクセル |
| 論理画面高 | 120ピクセル |
| 左パネルのx座標 | 0〜159 |
| 右パネルのx座標 | 160〜319 |

## 3. フレームバッファのメモリ形式

1ピクセルは、パレットを参照する符号なし8ビット整数で表します。
フレームバッファは行優先で隙間なく格納します。

```cpp
using ColorIndex = std::uint8_t;

constexpr std::size_t kScreenWidth = 320;
constexpr std::size_t kScreenHeight = 120;
constexpr std::size_t kFramebufferSize = kScreenWidth * kScreenHeight;

ColorIndex framebuffer[kFramebufferSize];
```

座標 `(x, y)` に対応するオフセットは、次の式で求めます。

```cpp
const std::size_t offset = y * kScreenWidth + x;
```

したがって、フレームバッファの容量は38,400バイトです。
行末のパディングは設けません。

フレームバッファは描画用と表示用の2面を確保し、ダブルバッファリングします。
2面の合計容量は76,800バイトです。

```cpp
ColorIndex framebuffers[2][kFramebufferSize];
```

描画完了後にフリップし、それまで描画用だった面を次の表示対象にします。描画中の面を
表示処理が読み取ってはいけません。

## 4. パレット形式

パレットは256色の配列です。各色は赤、緑、青の各チャンネルを符号なし8ビット整数で
表すRGB888形式とします。アルファチャンネルは持ちません。

```cpp
struct Rgb {
    std::uint8_t r;
    std::uint8_t g;
    std::uint8_t b;
};

constexpr std::size_t kPaletteSize = 256;
Rgb palette[kPaletteSize];
```

論理上の1エントリは3バイトです。構造体や配列を外部へ転送または保存する場合は、
コンパイラによるパディングを仮定せず、R、G、Bの順序を明示的に扱います。

パレットはダブルバッファリングせず、システム全体で1面だけ保持します。

フリップ時に、表示対象フレームバッファとその時点のパレットからLEDパネルへの転送に必要な
内部データを生成し、転送処理が使用する領域へ確定します。フリップ完了後のパレット変更が、
すでに確定したLED転送データへ影響してはいけません。

## 5. 予約色インデックス

次の色インデックスを予約します。

| インデックス | 用途 | RGB値 |
| ---: | --- | --- |
| 0 | スプライトの透明色、およびフレームバッファの背景黒 | `#000000` |
| 1 | スプライトで描画可能な完全な黒 | `#000000` |
| 2〜254 | 任意色 | パレット設定による |
| 255 | 完全な白 | `#ffffff` |

パレットのインデックス0、1、255は、それぞれ上表のRGB値に固定します。
公開APIはこれらの予約色の変更を拒否します。その他のパレットエントリは自動初期化を保証せず、
使用前にアプリケーションが明示的に設定します。

インデックス0と1はどちらも画面上では完全な黒ですが、スプライト描画時の意味が異なります。
黒い画素を持つスプライトでは、インデックス0ではなくインデックス1を使用します。

## 6. スプライトの透明色

スプライトの各画素も、フレームバッファと同じ8ビットの色インデックスで表します。

スプライト描画時にインデックス0の画素を読み取った場合は、フレームバッファへ書き込まず、
その位置にすでに存在する画素を維持します。アルファ合成は行いません。

```cpp
if (sprite_pixel != 0) {
    framebuffer[destination_offset] = sprite_pixel;
}
```

透明という意味はスプライト描画処理にだけ適用します。最終フレームバッファには透明の概念は
ありません。フレームバッファに格納されたインデックス0は、パレット0の完全な黒として
表示します。

フレームバッファの画素領域は自動的にクリアしません。描画を開始する前に、アプリケーションが
必要なRenderTargetと色を指定して明示的にクリアします。

## 7. プラットフォーム実装との境界

共通C++コードは、パレットと320×120ピクセルのフレームバッファを完成させるところまでを
担当します。

プラットフォーム固有の表示処理が担当するのは、以下の処理だけです。

- macOS版：パレットを参照して完成済みフレームバッファをSDLへ転送する
- マイコン版：完成済みフレームバッファとパレットからLED転送データを生成し、2枚のパネルへ送る

SDLの描画プリミティブを使用して、スプライト描画やフレームバッファ生成をmacOS専用に
再実装してはいけません。

対象環境向けにアセンブラ最適化を行う場合も、この仕様で定めたメモリ配置、色インデックス、
透明色処理、および描画結果を維持します。

## 8. 描画先ビュー

320×120ピクセルのフレームバッファは、全体を1画面として扱うほか、左右の160×120領域を
独立した2画面として扱えます。

```cpp
struct RenderTarget {
    std::uint8_t* pixels;
    std::int16_t originX;
    std::int16_t originY;
    std::uint16_t width;
    std::uint16_t height;
    std::uint16_t stride;
};
```

| 描画先 | ローカル座標 | フレームバッファ上の領域 |
| --- | --- | --- |
| 全画面 | x=0〜319、y=0〜119 | x=0〜319、y=0〜119 |
| 左画面 | x=0〜159、y=0〜119 | x=0〜159、y=0〜119 |
| 右画面 | x=0〜159、y=0〜119 | x=160〜319、y=0〜119 |

右画面のローカル座標 `(0, 0)` は、フレームバッファ上の座標 `(160, 0)` に対応します。
3つの描画先はいずれもストライドを320ピクセルとします。

すべての描画機能はRenderTargetのローカル座標を受け取り、そのwidthとheightでクリッピング
します。これにより、一方の画面を対象とした描画がもう一方へはみ出すことを防ぎます。
